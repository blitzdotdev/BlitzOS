# CLAUDE.md — BlitzOS (agent-os)

Guidance for Claude Code working in `agent-os/`. The root repo (`../CLAUDE.md`, teenybase) rules also apply.

## What this is

BlitzOS is an Electron macOS desktop for human+agent collaboration on live **surfaces** (browser windows, post-its, blitz.dev apps, agent-authored HTML). The human drives it, and so can an AI agent over the [agent-socket](https://agentsocket.dev) relay (paste a URL / "connect to blitz os", no MCP).

**Navigation model (the infinite canvas).** Both the Electron app and the browser/server preview run on one infinite canvas. Normal `desktop` mode homes the user on one bounded workspace area, but the human can pinch/scroll the empty canvas to zoom out and inspect surrounding/off-stage work without stealing gestures from live surfaces. A single Shift tap enters `canvas` Control Mode (the splayed all-stages view): a deliberate bird's-eye arrangement view where cards get a drag overlay and the empty canvas pans freely; a double Shift tap opens the workspace selector. The canvas is also the seam for multiple workspace areas and agent-driven follow/present behavior later. Rationale (HCI): the infinite plane is great for the agent (it teleports by id, no nav cost), while normal mode gives the human a stable frame until they intentionally zoom out. See `../plans/agent-os-desktop-architecture.md`.

**Agent-runtime model (why BlitzOS exists).** BlitzOS is an *OS for an agent*: it turns ANY connected agent (Claude Code today over agent-socket, any agent, a built-in chat client later) into an autonomous one with **zero per-task code**. The agent supplies intelligence; BlitzOS supplies the loop. Four parts: **syscalls** (the surface tools, the agent's hands), **perception** (a content-agnostic world stream, the agent's eyes), **a scheduler** (coalesced "moments" that *wake* the agent with a snapshot, the interrupt), and the **agent as swappable policy** (it decides significance and action). The whole point is **out-of-distribution generalization**: perception is dumb-but-rich and the agent decides what matters, so a new task (coach my chess, draft this email, summarize this PDF) needs no new BlitzOS code. **Never hand-build a per-task watch loop** — make perception and wake general, and let the agent's policy handle the task. See "Agent runtime" under Architecture.

Prototype stage; roadmap/backlog in `../plans/agent-os-desktop-architecture.md`, integration research in `../plans/agent-os-integrations-oauth.md`.

## Commands

```bash
npm run dev        # electron-vite dev (the GUI; macOS only)
npm run build      # compile main + preload + renderer to out/
npm run typecheck  # tsc --noEmit
npm run dist       # package BlitzOS.app (signed+notarized when ~/.zshrc Apple creds present) -> release/*.zip
```

There is **no display in CI / headless sandboxes**, so you cannot see the GUI. Verify behavior instead by: launching `npm run dev > /tmp/aos.log 2>&1`, then reading the log for `did-finish-load`, the printed control-API token, the agent-socket paste URL, and `[guest] loaded:` lines; and by driving the app via the control API or agent-socket tools (below) and checking `list_state`. Never claim the pixels look right, that's the user's to confirm.

## Stack & layout

electron-vite + React + TypeScript + zustand.

```
src/main/        Electron main (Node)
  index.ts         window, webview prefs, wires everything
  osActions.ts     THE control plane: create/move/close surfaces, getState. IPC to renderer.
  control-server.ts  localhost HTTP control API (local agent path)
  agentSocket.ts   connects to the agent-socket relay; tools -> osActions (remote agent path)
  cdp.ts           in-window control of `web` surfaces via CDP (webContents.debugger): click/type/key/read/screenshot
  integrations.ts  OAuth SSO registry + flows; tokenStore.ts = Keychain (safeStorage); oauth.ts = loopback code flow
  config.ts (inline in integrations.ts)  reads integrations.config.json
src/preload/index.ts   contextBridge api: onAction, sendState, onAgentSocketUrl, integrations.*
src/renderer/src/
  store.ts         zustand: transform, surfaces[], integrations[], actions
  App.tsx          desktop/canvas modes, renders surfaces + integration widgets + sidebar + toolbar
  components/SurfaceFrame.tsx  ONE frame, four renderers (web/app/srcdoc/native)
  components/BrowserNav.tsx    browser navbar of a web surface (back/forward/reload, address, bookmarks)
  components/NoteWidget.tsx    native 'note' = post-it
  components/{Sidebar,IntegrationWidget,ConnectPanel,PrimarySpace}.tsx
vendor/agent-socket-sdk/   vendored @agent-socket/sdk dist (see "Gotchas")
integrations.config.json   gitignored; OAuth client ids/secrets
```

## Architecture

- **Surface model (the core abstraction).** One descriptor `{id, kind, x, y, w, h, title, url?, html?, component?, props?}`; `SurfaceFrame` switches on `kind`:
  - `web` → a real BROWSER WINDOW: one main-owned `WebContentsView` PER TAB (`webcontents-view-host.ts`; the `<webview>` tag is gone — see the "Browser surfaces" gotcha). Real browsing context, framing-protections don't apply; each tab is a full process, so use sparingly.
  - `app` → `<iframe src>` (first-party blitz.dev apps you control).
  - `srcdoc` → `<iframe sandbox="allow-scripts" srcdoc>` (agent-authored HTML; isolated, no same-origin, no backend).
  - `native` → a built-in React component by `component` name (currently `note`).
  Pick the lightest kind that fits; agent-written code must be `srcdoc` (sandboxed), never `native`.
- **Control plane.** `osActions.ts` is the single source of truth for desktop mutations. Both the **local control server** (HTTP on 127.0.0.1) and **agent-socket** call it; it sends `os:action` IPC to the renderer, which applies it to the zustand store. The renderer pushes `os:state` back so `list_state` works. The agent↔OS contract is plain HTTP/JSON, **not MCP** (deliberate).
- **agent tools:** `create_surface`, `open_window` (web shortcut), `move_surface`, `update_surface`, `close_surface`, `go_to_primary`, `list_state`, `read_window` (read a web surface's DOM), `surface_control` (act inside a web surface), `/events` (the autonomy/watch loop), plus workspace/widget/`new_app`/`say`/`customize_widget` tools. **Defined ONCE in `src/main/os-tools.mjs`** (`makeOsTools(ops)`) — the single shared registry for ALL THREE transports: Electron relay (`agentSocket.ts`) + Electron localhost (`control-server.ts`) bind it via `electron-os-tools.ts` (`electronOps` → osActions IPC+CDP); the server (`preview/backend.mjs`) binds it via `serverOps` (SSE broadcast + headless Chromium). To add/change a tool, edit `os-tools.mjs` once. agentSocket.ts mints a paste URL surfaced via the in-app "Connect AI" button. The sandboxed-widget `blitz.tool` subset is the parallel shared registry `widget-tools.mjs` (`makeWidgetToolHandlers(ops)`), bound to the same `ops`.
- **In-window control (`surface_control`).** Acting *inside* a surface routes through `osActions.osControlSurface(id, action)`, keyed on `surface.kind`: **`web`** (a `WebContentsView` guest) → **CDP** via `webContents.debugger` (`cdp.ts`) — the only mechanism giving trusted input (`isTrusted`), no same-origin requirement, and off-screen reach; **`app`/`srcdoc`** (iframes) → cooperative `postMessage` (planned, not wired); **`native`** → store mutation. The debugger is single-client and attached lazily, then detached on idle/close so it never locks the user out of DevTools. CDP + perception always target the surface's ACTIVE tab: the view host re-registers it via `onActiveContent` on every tab switch AND every `dom-ready` (the re-register re-injects the perception sensors a navigation destroyed). **Security:** raw `eval` is allowed only on the localhost-bearer control server, **never over the relay** (`surface_control` rejects it).
- **Agent runtime (perception → moments → wake).** The autonomy half (see "Agent-runtime model" above). *Sensors* (`osActions.ts` `INJECT`) are injected into every web surface: input (key/click/input), **navigation**, **content change** (a `MutationObserver` — this is what catches drag interactions and async loads that fire no click), and **idle-after-activity**. A 350ms drain feeds raw signals to the *coalescer* (`events.ts`), which batches them into **moments** — framed snapshots `{trigger, signals, user[], snapshot, url, title}` — and emits one on a ~15s cadence OR immediately on a significant transition (nav / idle). `/events {since, wait}` long-polls moments; the agent runs ONE loop and is **woken** per moment, never per keystroke. Content-only churn (a running clock, an animation) refreshes the snapshot but does **not** wake the agent — only user signals do. **Generalization rule:** keep BlitzOS perception content-agnostic; the agent interprets significance and acts. No per-task detection (e.g. no "game over" logic) belongs in BlitzOS. **Transports:** agent-socket now (the agent runs the watch loop; BlitzOS drives by what it emits); a built-in chat client later gets more control (true push of turns, action gating).
- **Navigation modes (`store.mode`).** `desktop` is the normal view: scale-1 home on the current workspace area, with cursor-aware pinch zoom on the empty canvas and surface gestures left to the surface. Zoomed-out normal mode shows area frames; clicking an area homes into it. Canvas pan is captured as a short wheel-gesture burst once it starts, so crossing xterm/custom scrollers doesn't steal the pan; a fresh gesture that starts inside active scrollable surface content still belongs to that surface. `canvas` is Control Mode: a deliberate bird's-eye arrangement mode with the card drag overlay, background-drag pan, and area frames always visible. `go_to_primary` returns to the current area's home frame. The **dock** (Sidebar) + draggable **titlebar** move/focus windows in both.
- **Integrations.** OAuth SSO only (no token paste). `oauth.ts` runs a loopback authorization-code flow (system browser → `http://127.0.0.1:8723/callback` → token exchange in main → encrypted in Keychain via `safeStorage`). Each provider needs a one-time client id/secret in `integrations.config.json`.

## Gotchas / conventions

- **The SDK is ESM-only; Electron main is CJS.** So `@agent-socket/sdk` is **bundled** into main via `externalizeDepsPlugin({ exclude: ['@agent-socket/sdk'] })` in `electron.vite.config.ts`, not `require()`d. `ws` stays external.
- **The SDK is vendored** at `vendor/agent-socket-sdk/` (its own repo is `~/agent-socket`, github.com/repalash/agent-socket). To update: rebuild it (`cd ~/agent-socket/sdk && npm run build`) then re-copy `dist` + `package.json` into `vendor/agent-socket-sdk/`.
- **Never commit secrets.** `integrations.config.json` (real client secrets) is gitignored; so are `node_modules/`, `out/`. Verify before any commit.
- **OAuth redirect is fixed:** `http://127.0.0.1:8723/callback` — register it on every provider's OAuth app. Use `127.0.0.1`, not `localhost` (providers treat them as different hosts).
- **Layout IS persisted** — the active workspace folder is the datasource: `<dir>/.blitzos/workspace.json` holds `{ camera, mode, stack, stageCount, nodes[] }` and each surface's content is a file in the folder. Writes are durably flushed at call time (authoritative-on-write), so surface ids + layout survive a restart and rehydrate on boot. (Runtime-only surfaces — chat/activity/terminal/runtime/inbox/unlock — are reconstructed, not persisted as nodes.)
- **Off-screen liveness:** `backgroundThrottling: false` on the window and every browser-tab view, so panned-away surfaces keep running for the agent.
- **Browser surfaces (`web`) are real browser windows** — one main-owned `WebContentsView` PER TAB (`webcontents-view-host.ts`; min-browser architecture, references cloned in `../.repos/{min,browser-base}`; the `<webview>` tag is deleted). The renderer DECLARES `{tabs, active}` (the terminal-style `surface.tabs`, materialized lazily from the legacy `{url}` shape) + reports the body rect per RAF; main reconciles live views and PUSHES per-tab page state (`url/title/favicon/loading/canGoBack/canGoForward`) on `os:web-tab` — chrome never polls. Chrome = the standard window bar + tab strip + `BrowserNav.tsx` (back/forward/reload via `navigationHistory`, address bar with a per-tab draft that nav events may only clobber while unfocused, star + bookmarks dropdown — plain DOM that simply paints over the live page). A link-disposition popup becomes a NEW TAB of its surface (`onOpenTab`); other popup plans unchanged (`guest-capabilities.ts`). An url-less tab still loads `about:blank` — a never-committed webContents queues every `executeJavaScript` forever (hung CDP evals, perception-drain listener pile-up). Bookmarks are machine-global in the root journal (`readBookmarks`/`toggleBookmark`, workspace.mjs). Tabs persist in the `.weblink` (`{url, tabs:[{id,title,url}], activeTab}`; top-level `url` = the ACTIVE tab, so the agent contract + `account_hint` are unchanged). Three races are guarded in the host; keep them: a StrictMode remount must never orphan a view (identity-checked `destroyed` drops + an 80ms deferred close that a re-sync cancels), a pre-commit `did-start-loading` must never push `url:''` (it cleared persisted tabs), and once `before-quit` fires the host goes SILENT (teardown `destroyed` events otherwise close store tabs mid-final-flush) with every renderer send routed through the destruction-guarded `sendToRenderer` (a bare `getWin()?.webContents.send` THROWS on a destroyed window).
- **The sandwich compositor** (`sandwich.ts`, `plans/blitzos-sandwich-compositor.md`): a native view paints above ALL renderer DOM in its window, so BlitzOS runs as TWO congruent windows — L0 "pages" (bottom: every page WebContentsView; hidden from Mission Control) and L1 "UI" (top, `transparent:true` + hiddenInset traffic lights: the entire renderer; `mainWindow` in index.ts IS L1). Each browser body is a transparent HOLE (`.webcontents-host`; the `html/body/#root` base is transparent too — the desktop's canvas color is painted by L0's `backgroundColor`, NOT by DOM). Stacking, all live, no freezing: DOM over page = just paints (physically above); page over DOM = the lower DOM surface gets a CSS `clip-path` hole for the page rect (`pageHolesClip` in SurfaceFrame + `sceneryClip` in PrimarySpace for the translucent stage tint; world coords, camera-free); page over page = child-view order in L0; DOM over DOM = z-index. **The pair is PARENTED (`ui.setParentWindow(pages)`) and that is LOAD-BEARING:** macOS occlusion-culls a standalone window that another window fully covers — the views in L0 simply stop compositing (bisect-verified: unparented = blank holes, parented = live pages) — and an attached group is exempt AND cannot be interleaved by foreign windows. The cost: the attached child must never be moved/resized directly (macOS child glue is one-way and feedback-loops), so the titlebar drag is MANUAL (App.tsx streams pointer deltas over `os:shell-drag`; main moves the PARENT and the child follows — never add `-webkit-app-region` back), the UI window is non-resizable for now, the yellow minimize light is dead on the child (macOS), and fullscreen rides the parent (children join its Space, bounds-synced on `enter-full-screen`). Input: L1 owns all mouse; hole events forward via `os:page-input` → `sendInputEvent` (wheel deltas NEGATED — Electron scrolls down on negative deltaY); keyboard is NATIVE via focus handoff (`os:page-focus` → the pages window becomes key; the attached child stays above it; any UI pointerdown → `os:ui-focus` back); page cursors mirror via `cursor-changed` → CSS cursor on the hole. THE GLASS RULE (the bug that ate a day): a browser hole must be ALPHA-0 ALL THE WAY DOWN the DOM — a `background:transparent` div does NOT punch through opaque DOM below it, so any full-bleed opaque layer under windows (it was `.bg`, the canvas pan-target; neutralized via `body.sandwich .bg`) silently blanks every hole with its own pixels, which reads exactly like "views not compositing". When debugging holes, check the element stack under the hole FIRST (document.elementsFromPoint), before suspecting the compositor. CLIP RULES: holes are cut with `path()` SUBPATHS via `holesPath()` (SurfaceFrame) — never a single even-odd `polygon()` (ring-connector pixels antialias into hairline ghost outlines of the clipped element); the outer ring is padded past the element, holes wind OPPOSITE the outer ring (the `evenodd` keyword inside path() is not parsed by every Chromium), and full coverage returns the `HIDE` sentinel (hide the element; a degenerate clip ghosts its outline). The desktop base `.bg` STAYS OPAQUE with screen-space holes (App.tsx bgClip) — a fully transparent base made every window box-shadow composite against glass (dark pooling fringes). Keyboard handoff is CONDITIONAL: after a page click, main probes document.activeElement and only an editable target moves the key window (flipping it every click grays the UI chrome). Known warts: telemetry/thumbnail `capturePage` of L1 no longer includes pages (holes capture transparent); L0's backdrop color is static (`#e9e9e7`) — dark theme needs a recolor IPC; shell resize + minimize are follow-ups.
- **Popup policy** (`popup-policy.mjs` + `attachGuestWindowPolicy`, attached per tab view): `window.open` must be answered or sites fall back to `top.location = url` and hijack the surface. Plans: sized `features` → real window (OAuth), `about:blank` → hidden child, link disposition → new TAB in the owning surface, else deny + swallow the `will-navigate` hijack. Never "allow as hidden window" for URLs that fail to load top-level — that create/fail/destroy churn use-after-free'd Electron 31's guest-view manager (the 2026-06-09 SIGSEGV).
- Relay default is `agentsocket.dev` (`aisocket.dev` = same Worker); override with `AGENT_SOCKET_RELAY`.
- **Stage slot desktop** (`plans/blitzos-stage-slot-desktop.md`, RE'd from macOS): the user-facing desktop (the workspace AREA their desktop-mode camera frames) is a **slot lattice** (`stage-core.mjs`, pure, shared all transports — 180pt tiles, 8pt card inset, spans s/m/l/xl/tall/xxl); slotted tiles **never overlap, never reflow** (occupied spans are simply never offered). Agents place by **slot, never pixels**: `place_widget {size, near?}` / `bring_to_stage` / `send_backstage`. **There is NO separate backstage zone**: off-stage = the open infinite canvas outside the stage — web/app surfaces auto-PARK below the stage (`parkOffstage`), naturally off-screen at the user's scale-1 frame and visible when they zoom out (control mode); `isOffstage` is computed geometrically, nothing is hidden via CSS. A non-pinned budget (`STAGE_BUDGET` S-units) returns `stage_full` so the agent evicts/queues instead of overflowing attention. **Layered desktop:** tiles + file/folder icons are the DESKTOP layer; free-form windows FLOAT ABOVE them (z-bands: tiles/icons → free windows +500k → focus +1.5M → pinned +2M) — free windows reserve NO lattice cells (a tile snaps in beneath them, like a window over macOS desktop widgets), and an INVISIBLE tile (minimized/foldered) releases its cells, re-placing via the placer on restore if its span got taken. Human gestures: tile drag = float + outline ghost (reuses snap-preview) + spring snap (a dragged tile lifts above the window band); **⌘-drag pops a tile off the lattice**; the **window-bar grid toggle** (⊞/⤢) or **⌘T** snaps any window INTO the nearest free span / pops a tile OUT restoring its pre-slot size; **⇧⌘T** cycles a tile's size (on a free window it snaps in first) through SIZE_ORDER (anchored at its cell when the new span fits) (`preSnap` — slot sizes must never be a one-way door, e.g. the chat). File/dir tiles are the **fluid layer** (`flowFiles`) parting around tiles + the drag ghost. x/y/w/h stay the rendering+persistence truth, derived from `slot` on hydrate/resize (`stageFields` persists `slot`/`slotStage` only). Tests: `node scripts/test-stage-core.mjs` + `node scripts/test-stage-e2e.mjs`.

- **Stage splay lattice** (`plans/blitzos-stage-splay-lattice.md`): stages sit on macOS-style RAGGED CENTERED ROWS, not one endless row — ported from GNOME's `UnalignedLayoutStrategy` (the reverse-engineered Mission Control lineage; cloned in `../.repos/gnome-shell`), in the shared `stages-core.mjs` (`splayRows`/`splayLayout`/`orderedStageRect`/`stageOfPoint`). **World-REAL:** a stage's cell is real canvas geometry; `stageOrder` (persisted in workspace.json, identity default, self-healing) is the reading order; dragging a stage label in the splay insertion-reflows the order and `applyStageOrder` translates every member surface with its stage. **Membership is ONE shared rule** (`surfaceStage`: slotStage → chat agent → geometric center) used by both the reorder AND the per-stage Sidebar dock (the dock shows only the current stage's surfaces). The "create new stage" placeholder is a permanent layout citizen (slot N — world positions never depend on the splay being open), park bands live in each stage's cell gutter (`parkBandRect`; `parkOffstage` targets it), the splay camera bbox-fits the whole lattice, and a reorder stamps `bulkAt` on the os:state push so perception treats it as ONE gesture, not N window-moves. Tests: `node scripts/test-stage-splay-core.mjs`.
- **Computer Use helper** (`plans/blitzos-computer-use-helper.md`, RE'd from Codex in `plans/codex-computer-use-tcc-reference.md`): Accessibility + Screen Recording TCC grants would force the GRANTED app to quit+reopen — fatal mid-session. So they DON'T live on BlitzOS: a separate Developer-ID-signed native helper **`BlitzComputerUse.app`** (`dev.blitz.os.computeruse`, `LSUIElement`, `native/computer-use-helper/`, built+signed by `scripts/dist-mac.sh`) HOLDS them. The load-bearing trick: the helper is launched via **LaunchServices** (`open -n`), making it its OWN responsible process with its OWN TCC identity — a child spawned by us would inherit BlitzOS's identity (same reason the scan child inherits our FDA). IPC = a Unix socket BlitzOS owns + newline-JSON (no inherited stdio across a LaunchServices launch); liveness = the socket. "Quit & reopen for the grant to take effect" = relaunch the HELPER (`relaunchForGrant`), BlitzOS untouched. Lifecycle in `src/main/computer-use-helper.ts` (install to `~/Library/Application Support/BlitzOS` for a stable bundle the user's grant sticks to, supervise, RPC); the pre-board Accessibility/Screen steps drive it (request lists it, the drag tile drags the HELPER bundle, poll its TCC, relaunch on grant). FDA stays on BlitzOS (read live by the scan child, no restart). Tests: `node scripts/test-computer-use-helper.mjs`. Dev caveat: TCC identity separation is only real in a signed packaged build; dev verifies build/launch/socket/relaunch mechanics.
- **Kernel fault model:** one instance per machine (`requestSingleInstanceLock`; a second launch focuses the first). `<root>/.blitzos/state.json` is the runtime journal: boot returns to the **last-active workspace** (a `BLITZ_WORKSPACE` pin overrides), and a boot after an unclean shutdown announces the crash to the human (chat line) + the agent (`trigger:'system'` moment, kind `crash`), enriched from macOS DiagnosticReports when possible. `markClean` runs LAST in `before-quit`/`gracefulExit` — "clean" means state flushed first. Two hosts on one root are detected via the journal's pid+heartbeat (loud warning today; refuse-enforcement pending decision). Tests: `node scripts/test-root-state.mjs`.
- **Effect-verified syscalls (`ok` = it happened + is durable):** `surface_control` click resolves the **visible, hit-testable** match (not the first `querySelector` hit — sites keep hidden twins; a selector matching only hidden/covered elements is a loud error), and click/type/key return an `effect` (typed value back, url/dom change) so the agent verifies in-band. Surface mutations are **authoritative-on-write**: main/serverOps apply create/update/move/close to `cached`/`osState` immediately (a `pendingCreates` set covers the pre-renderer-echo race), return `ok:false` for an unknown id (no more silent no-ops), and force a **durable flush at call time** on create/update/close so an `ok` ack survives a crash (the gap that lost a note). Tests: `node preview/test-control-effects.mjs` (needs CHROMIUM/Chrome).
- **Per-surface account hint** (item 7): `list_state` tags a `web` surface with `account_hint {provider, label, verify:true}` when a CONNECTED integration's web hosts match its URL (content-agnostic `PROVIDER_WEB_HOSTS` suffix-match in `os-tools.mjs`; no per-site DOM logic). It's a HINT an account exists, NOT proof of the surface's signed-in account (a webview can be a different account than the OAuth integration) — the doctrine tells the agent to `read_window` to verify before acting AS it. Reliable per-site identity extraction was deliberately NOT built (can't be content-agnostic).
- **Restart continuity / cross-workspace addressing** (item 4): surface ids already survive a restart (item 2's durable flush persists the workspace.json node at call time — verified). The remaining gap, **cross-workspace addressing**, is `findSurfaceWorkspace`/`relocateSurface` (workspace.mjs) + host `locateSurface`/`bringSurfaceHere`: an op on an id NOT in the active workspace no longer dead-ends — `move_surface` BRINGS the window into the active workspace (its content file is moved across folders, **id preserved**), while update/close/read return a navigable error naming the workspace ("move_surface it here, or switch_workspace"). The agent decides: just-this-window → move; whole-desktop → switch. Tests: `node scripts/test-cross-workspace.mjs`.
- **Guest capability contract** (`guest-capabilities.ts` + pure `popup-policy.mjs`): a web guest's browser-initiated escapes are handled by **what kind of action it is, never which site** (the old `accounts.google.com`/`contacts.google.com` popup regexes are deleted). Popups: sized `features`→real window (generalizes OAuth), `about:blank`→hidden child, link-disposition→new surface, else→deny+swallow the `top.location` hijack. Downloads→stream into the active workspace folder (→ a file tile).
