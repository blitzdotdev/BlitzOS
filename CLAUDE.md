# CLAUDE.md — BlitzOS (agent-os)

Guidance for Claude Code working in `agent-os/`. The root repo (`../CLAUDE.md`, teenybase) rules also apply.

## What this is

BlitzOS is an Electron macOS desktop for human+agent collaboration on live **surfaces** (browser windows, post-its, blitz.dev apps, agent-authored HTML). The human drives it, and so can an AI agent over the [agent-socket](https://agentsocket.dev) relay (paste a URL / "connect to blitz os", no MCP).

**Navigation model (single canvas, one home).** Both the Electron app and the browser/server preview run on one infinite canvas with exactly ONE bounded region, **home** (the computed scale-1 frame, `homeRect`/`homeTransform`; no saved camera). Home holds the slot lattice; off-home is open canvas where web/app windows park. The human can pinch/scroll the empty canvas to zoom out and inspect off-home work without stealing gestures from live surfaces. A single Shift tap is the **freeze toggle** (`locked` gates pan/zoom so clicks stay with surfaces); a double Shift tap **flies home and freezes**; **ESC** opens the workspace switcher. Workspaces replace the old multi-stage model. Rationale (HCI): the infinite plane is great for the agent (it teleports by id, no nav cost), while the home frame gives the human a stable frame until they intentionally zoom out. See `../plans/blitzos-single-canvas-navigation.md` and `../plans/agent-os-desktop-architecture.md`.

**Agent-runtime model (why BlitzOS exists).** BlitzOS is an *OS for an agent*: it turns ANY connected agent (Claude Code today over agent-socket, any agent, a built-in chat client later) into an autonomous one with **zero per-task code**. The agent supplies intelligence; BlitzOS supplies the loop. Four parts: **syscalls** (the surface tools, the agent's hands), **perception** (a content-agnostic world stream, the agent's eyes), **a scheduler** (coalesced "moments" that *wake* the agent with a snapshot, the interrupt), and the **agent as swappable policy** (it decides significance and action). The whole point is **out-of-distribution generalization**: perception is dumb-but-rich and the agent decides what matters, so a new task (coach my chess, draft this email, summarize this PDF) needs no new BlitzOS code. **Never hand-build a per-task watch loop** — make perception and wake general, and let the agent's policy handle the task. See "Agent runtime" under Architecture.

Prototype stage; roadmap/backlog in `../plans/agent-os-desktop-architecture.md`, integration research in `../plans/agent-os-integrations-oauth.md`.

## Commands

```bash
npm run dev        # electron-vite dev (the GUI; macOS only)
npm run build      # compile main + preload + renderer to out/
npm run typecheck  # tsc --noEmit
npm run dist       # package BlitzOS.app (signed+notarized when ~/.zshrc Apple creds present) -> release/*.zip
```

There is **no display in CI / headless sandboxes**, so you cannot see the GUI. Verify behavior instead by: launching `npm run dev > /tmp/aos.log 2>&1`, then reading the log for `did-finish-load`, the printed control-API token, the agent-socket paste URL, and `[guest] loaded:` lines; and by driving the app via the control API or agent-socket tools (below) and checking `list_state`. Never claim the pixels look right, that's the user's to confirm.

## Stack & layout

electron-vite + React + TypeScript + zustand.

```
src/main/        Electron main (Node)
  index.ts         window, webview prefs, wires everything
  osActions.ts     THE control plane: create/move/close surfaces, getState. IPC to renderer.
  control-server.ts  localhost HTTP control API (local agent path)
  agentSocket.ts   connects to the agent-socket relay; tools -> osActions (remote agent path)
  cdp.ts           in-window control of `web` surfaces via CDP (webContents.debugger): click/type/key/read/screenshot
src/preload/index.ts   contextBridge api: onAction, sendState, onAgentSocketUrl
src/renderer/src/
  store.ts         zustand: transform, surfaces[], actions
  App.tsx          desktop/canvas modes, renders surfaces + sidebar + toolbar
  components/SurfaceFrame.tsx  ONE frame, four renderers (web/app/srcdoc/native)
  components/BrowserNav.tsx    browser navbar of a web surface (back/forward/reload, address, bookmarks)
  components/NoteWidget.tsx    native 'note' = post-it
  components/{Sidebar,PrimarySpace}.tsx
vendor/agent-socket-sdk/   vendored @agent-socket/sdk dist (see "Gotchas")
```

## Architecture

- **Surface model (the core abstraction).** One descriptor `{id, kind, x, y, w, h, title, url?, html?, component?, props?}`; `SurfaceFrame` switches on `kind`:
  - `web` → a real in-DOM `<webview>` guest PER TAB (`components/WebTabView.tsx`): a separate-process browsing context that is an ordinary DOM child of its frame, so it moves/stacks/clips with the window like any DOM — no compositor. Real browsing context, framing-protections don't apply; each tab is a full process, so use sparingly. (The two-window "sandwich" compositor + per-tab `WebContentsView` host were removed; that architecture is preserved on branch `content-view-compositor` — see the "Browser surfaces" gotcha.)
  - `app` → `<iframe src>` (first-party blitz.dev apps you control).
  - `srcdoc` → `<iframe sandbox="allow-scripts" srcdoc>` (agent-authored HTML; isolated, no same-origin, no backend).
  - `native` → a built-in React component by `component` name (currently `note`).
  Pick the lightest kind that fits; agent-written code must be `srcdoc` (sandboxed), never `native`.
- **Control plane.** `osActions.ts` is the single source of truth for desktop mutations. Both the **local control server** (HTTP on 127.0.0.1) and **agent-socket** call it; it sends `os:action` IPC to the renderer, which applies it to the zustand store. The renderer pushes `os:state` back so `list_state` works. The agent↔OS contract is plain HTTP/JSON, **not MCP** (deliberate).
- **agent tools:** `create_surface`, `open_window` (web shortcut), `move_surface`, `update_surface`, `close_surface`, `go_to_primary`, `list_state`, `read_window` (read a web surface's DOM), `surface_control` (act inside a web surface), `/events` (the autonomy/watch loop), plus workspace/widget/`new_app`/`say`/`customize_widget` tools. **Defined ONCE in `src/main/os-tools.mjs`** (`makeOsTools(ops)`) — the single shared registry for ALL THREE transports: Electron relay (`agentSocket.ts`) + Electron localhost (`control-server.ts`) bind it via `electron-os-tools.ts` (`electronOps` → osActions IPC+CDP); the server (`preview/backend.mjs`) binds it via `serverOps` (SSE broadcast + headless Chromium). To add/change a tool, edit `os-tools.mjs` once. agentSocket.ts mints a paste URL surfaced via the in-app "Connect AI" button. The sandboxed-widget `blitz.tool` subset is the parallel shared registry `widget-tools.mjs` (`makeWidgetToolHandlers(ops)`), bound to the same `ops`.
- **In-window control (`surface_control`).** Acting *inside* a surface routes through `osActions.osControlSurface(id, action)`, keyed on `surface.kind`: **`web`** (a `WebContentsView` guest) → **CDP** via `webContents.debugger` (`cdp.ts`) — the only mechanism giving trusted input (`isTrusted`), no same-origin requirement, and off-screen reach; **`app`/`srcdoc`** (iframes) → cooperative `postMessage` (planned, not wired); **`native`** → store mutation. The debugger is single-client and attached lazily, then detached on idle/close so it never locks the user out of DevTools. CDP + perception always target the surface's ACTIVE tab: the view host re-registers it via `onActiveContent` on every tab switch AND every `dom-ready` (the re-register re-injects the perception sensors a navigation destroyed). **Security:** raw `eval` is allowed only on the localhost-bearer control server, **never over the relay** (`surface_control` rejects it).
- **Agent runtime (perception → moments → wake).** The autonomy half (see "Agent-runtime model" above). *Sensors* (`osActions.ts` `INJECT`) are injected into every web surface: input (key/click/input), **navigation**, **content change** (a `MutationObserver` — this is what catches drag interactions and async loads that fire no click), and **idle-after-activity**. A 350ms drain feeds raw signals to the *coalescer* (`events.ts`), which batches them into **moments** — framed snapshots `{trigger, signals, user[], snapshot, url, title}` — and emits one on a ~15s cadence OR immediately on a significant transition (nav / idle). `/events {since, wait}` long-polls moments; the agent runs ONE loop and is **woken** per moment, never per keystroke. Content-only churn (a running clock, an animation) refreshes the snapshot but does **not** wake the agent — only user signals do. **Generalization rule:** keep BlitzOS perception content-agnostic; the agent interprets significance and acts. No per-task detection (e.g. no "game over" logic) belongs in BlitzOS. **Transports:** agent-socket now (the agent runs the watch loop; BlitzOS drives by what it emits); a built-in chat client later gets more control (true push of turns, action gating).
- **Navigation (single home + freeze).** `store.mode` is pinned `'desktop'` (the field is dormant, scheduled for deletion). The normal view is scale-1 on **home** (`homeRect`/`homeTransform`), with cursor-aware pinch zoom on the empty canvas and surface gestures left to the surface. Zooming out reveals off-home work parked around home (web/app windows auto-park below it); there is no separate stage/control mode and no splay. Canvas pan is captured as a short wheel-gesture burst once it starts, so crossing xterm/custom scrollers doesn't steal the pan; a fresh gesture that starts inside active scrollable surface content still belongs to that surface. A single Shift tap toggles the **freeze lock** (`store.locked` gates pan/zoom, frozen means clicks reach surface content); a double Shift tap flies home and freezes; **ESC** opens the workspace switcher. `go_to_primary` flies to the home frame. The **dock** (Sidebar) + draggable **titlebar** move/focus windows.
- **Auth is browser-first — there is NO OAuth/integrations subsystem** (removed 2026-06-16, commit 629b40d). The agent works the user's tools ONLY through their LOGGED-IN web surfaces (`read_window` / `surface_control`); the browser session (`persist:agentos` partition) is the real, user-maintained auth. The old parallel path — provider OAuth tokens, the `provider_call` tool, the `blitz.data` widget fetch, `account_hint` — was removed as a footgun (an expired token nagged the user to "reconnect Gmail" while they were already signed in there). Sized-`features` login popups still open as real windows (generic popup policy, see Guest capability contract).

## Gotchas / conventions

- **Design quality bar.** UI you build must match the real BlitzOS design, not generic AI slop. Read the design tokens and match them, and build widgets you would proudly ship in the default library. When fixing a visual flaw, fix every instance (all corners, all overflow, all intersecting chrome), not only the one called out.
- **The SDK is ESM-only; Electron main is CJS.** So `@agent-socket/sdk` is **bundled** into main via `externalizeDepsPlugin({ exclude: ['@agent-socket/sdk'] })` in `electron.vite.config.ts`, not `require()`d. `ws` stays external.
- **The SDK is vendored** at `vendor/agent-socket-sdk/` (its own repo is `~/agent-socket`, github.com/repalash/agent-socket). To update: rebuild it (`cd ~/agent-socket/sdk && npm run build`) then re-copy `dist` + `package.json` into `vendor/agent-socket-sdk/`.
- **Never commit secrets.** `node_modules/`, `out/`, and `release/` are gitignored. Verify before any commit.
- **Layout IS persisted** — the active workspace folder is the datasource: `<dir>/.blitzos/workspace.json` holds `{ camera, mode, stack, nodes[] }` and each surface's content is a file in the folder. Writes are durably flushed at call time (authoritative-on-write), so surface ids + layout survive a restart and rehydrate on boot. (Runtime-only surfaces — chat/activity/terminal/runtime/inbox/unlock — are reconstructed, not persisted as nodes.)
- **Off-screen liveness:** `backgroundThrottling: false` on the window and every browser-tab view, so panned-away surfaces keep running for the agent.
- **Browser surfaces (`web`) are in-DOM `<webview>` guests** — one `<webview>` PER TAB (`components/WebTabView.tsx`), a separate-process browsing context that is an ordinary DOM child of the frame, so it moves/stacks/clips with the window like any DOM — NO geometry sync, no compositor (the whole point of going back to `<webview>`). The renderer owns `{tabs, active}` (`surface.tabs`, materialized lazily from the legacy `{url}`). Tabs are **lazy-then-live**: a tab gets a live `<webview>` once it has been active and then STAYS mounted-but-hidden (`webview.wt-inactive`), so background tabs keep their page/scroll/form/history (`SurfaceFrame` `materializedTabs`). Per-tab page state (`url/title/favicon/loading/canGoBack/canGoForward`) is mirrored RENDERER-LOCALLY from each `<webview>`'s own events (`did-navigate`/`did-start|stop-loading`/`page-favicon-updated`) via `applyWebTab` — the old main→renderer `os:web-tab` push is GONE. The ACTIVE tab reports its guest WebContents id to main (`registerWebview` → `os:webview` → `registerLiveWebContent`, osActions) so the agent's read/control (CDP, `cdp.ts`) + perception (INJECT/DRAIN, re-injected on each `dom-ready`) reach the live page; it also carries `data-sid` so `BrowserNav.tsx` (back/forward/reload via the `<webview>` element methods, address bar with a per-tab draft) + radial-over-web detection find it. Chrome = window bar + tab strip + `BrowserNav.tsx` (star + bookmarks dropdown — plain DOM). A link-disposition popup becomes a new tab (`allowpopups` + `guest-capabilities.ts` policy, attached per guest via `web-contents-created` in index.ts). Bookmarks are machine-global in the root journal (`readBookmarks`/`toggleBookmark`). Tabs persist in the `.weblink` (`{url, tabs:[{id,title,url}], activeTab}`; top-level `url` = ACTIVE tab, agent contract unchanged). **HTML5 `<video>` fullscreen** works NATIVELY on the `<webview>` element (the element enters the Fullscreen API); `WebTabView` mirrors `enter/leave-html-full-screen` into `store.pageFullscreenId` for agent/state awareness (no sandwich chrome-baring needed).
- **The sandwich compositor was REMOVED from master** (web is in-DOM `<webview>` now — see "Browser surfaces"). BlitzOS is ONE plain `BrowserWindow` (`index.ts`); the titlebar is a native `-webkit-app-region` drag with native `titleBarStyle:'hiddenInset'` traffic lights (no custom `os:shell-*` controls, no two-window pair). The deleted design — two congruent windows (L0 "pages" `WebContentsView`s + L1 transparent "UI" DOM), clip-path HOLES (`pageHolesClip`/`holesPath`/the GLASS RULE), `os:page-input`/`os:page-focus`/`os:ui-focus`/`os:shell-drag`, custom DOM traffic-lights, `os:web-fullscreen` page-fullscreen — is gone from `sandwich.ts`/`webcontents-view-host.ts` (both deleted). **Why it existed:** a native `WebContentsView` can't take a z-index *between* DOM elements, so showing DOM *over* a page needed either clip-holes (the sandwich) or stacked OS windows (a later "layered windows, no holes" compositor). Both approaches — and `plans/blitzos-sandwich-compositor.md` — are preserved on branch **`content-view-compositor`**. `<webview>` is a DOM node, so z-index/drag/clip are free; that simplicity is why master uses it. (Tradeoff: a `<webview>` is deprecated-but-supported in Electron; the layered-windows branch is the fallback if it bites.)
- **The notch (dynamic island) overlay** (`notch-overlay.ts`) — the ONE exception to "no custom `os:shell-*`/overlay" above. When notch-gated, the single `BrowserWindow` is reconfigured as a transparent, all-Spaces, full-display overlay (the notch-essential bits salvaged from the removed sandwich: the overlay window options + the `os:notch-interactive` click-through toggle the renderer drives on hover). The renderer clips `#root-canvas` to the notch shape and grows the clip to reveal the live canvas; the physical handle is a body-portal pill (App.tsx) z-stacked above the island chassis so it stays clickable. In this mode the native traffic lights are hidden and App.tsx draws its own (`os:shell-{close,minimize,fullscreen}` → the single window; fullscreen is a no-op because an all-Spaces overlay can't native-fullscreen — the notch's "fullscreen" IS the clip-grow). Web surfaces stay in-DOM `<webview>` (no compositor). In normal (non-notch) mode it is master's plain hiddenInset window.
- **Popup policy** (`popup-policy.mjs` + `attachGuestWindowPolicy`, attached per tab view): `window.open` must be answered or sites fall back to `top.location = url` and hijack the surface. Plans: sized `features` → real window (OAuth), `about:blank` → hidden child, link disposition → new TAB in the owning surface, else deny + swallow the `will-navigate` hijack. Never "allow as hidden window" for URLs that fail to load top-level — that create/fail/destroy churn use-after-free'd Electron 31's guest-view manager (the 2026-06-09 SIGSEGV).
- Relay default is `agentsocket.dev` (`aisocket.dev` = same Worker); override with `AGENT_SOCKET_RELAY`.
- **Home lattice** (`plans/blitzos-single-canvas-navigation.md`, RE'd from macOS desktop widgets): each workspace has ONE bounded **slot lattice** that lives only at **home** (`homeRect`, the computed scale-1 frame; `stages-core.mjs` exports just `{DEFAULT_VP, homeRect, parkBandRect}`). The lattice itself is `stage-core.mjs` (pure, shared all transports, no stage/order/count params; 180pt tiles, 8pt card inset, spans s/m/l/xl/tall/xxl); slotted tiles **never overlap, never reflow** (occupied spans are simply never offered). Agents place by **slot, never pixels**: `place_widget {size, near?}` / `bring_home` / `send_offscreen`. **There is NO separate off-home zone**: off-home = the open infinite canvas outside home, so web/app surfaces auto-PARK below home (`parkOffstage` → `parkBandRect`), naturally off-screen at the user's scale-1 frame and visible when they zoom out; `isOffstage` is computed geometrically (surface center outside `homeRect`), nothing is hidden via CSS. A non-pinned budget (`HOME_BUDGET` S-units) returns `home_full` so the agent evicts/queues instead of overflowing attention. **Layered desktop:** tiles + file/folder icons are the DESKTOP layer; free-form windows FLOAT ABOVE them (z-bands: tiles/icons → free windows +500k → focus +1.5M → pinned +2M) — free windows reserve NO lattice cells (a tile snaps in beneath them, like a window over macOS desktop widgets), and an INVISIBLE tile (minimized/foldered) releases its cells, re-placing via the placer on restore if its span got taken. Human gestures: tile drag = float + outline ghost (reuses snap-preview) + spring snap (a dragged tile lifts above the window band); **⌘-drag pops a tile off the lattice**; the **window-bar grid toggle** (⊞/⤢) or **⌘T** snaps any window INTO the nearest free span / pops a tile OUT restoring its pre-slot size; **⇧⌘T** cycles a tile's size (on a free window it snaps in first) through SIZE_ORDER (anchored at its cell when the new span fits) (`preSnap` — slot sizes must never be a one-way door, e.g. the chat). File/dir tiles are the **fluid layer** (`flowFiles`) parting around tiles + the drag ghost. x/y/w/h stay the rendering+persistence truth, derived from `slot` on hydrate/resize (`stageFields` persists `slot` only). Tests: `node scripts/tests/test-stage-core.mjs` + `node scripts/tests/test-stage-e2e.mjs`.
- **Computer Use helper** (`plans/blitzos-computer-use-helper.md`, RE'd from Codex in `plans/codex-computer-use-tcc-reference.md`): Accessibility + Screen Recording TCC grants would force the GRANTED app to quit+reopen — fatal mid-session. So they DON'T live on BlitzOS: a separate Developer-ID-signed native helper **`BlitzComputerUse.app`** (`dev.blitz.os.computeruse`, `LSUIElement`, `native/computer-use-helper/`, built+signed by `scripts/dist-mac.sh`) HOLDS them. The load-bearing trick: the helper is launched via **LaunchServices** (`open -n`), making it its OWN responsible process with its OWN TCC identity — a child spawned by us would inherit BlitzOS's identity (same reason the scan child inherits our FDA). IPC = a Unix socket BlitzOS owns + newline-JSON (no inherited stdio across a LaunchServices launch); liveness = the socket. "Quit & reopen for the grant to take effect" = relaunch the HELPER (`relaunchForGrant`), BlitzOS untouched. Lifecycle in `src/main/computer-use-helper.ts` (install to `~/Library/Application Support/BlitzOS` for a stable bundle the user's grant sticks to, supervise, RPC); the pre-board Accessibility/Screen steps drive it (request lists it, the drag tile drags the HELPER bundle, poll its TCC, relaunch on grant). FDA stays on BlitzOS (read live by the scan child, no restart). Tests: `node scripts/test-computer-use-helper.mjs`. Dev caveat: TCC identity separation is only real in a signed packaged build; dev verifies build/launch/socket/relaunch mechanics.
- **Kernel fault model:** one instance per machine (`requestSingleInstanceLock`; a second launch focuses the first). `<root>/.blitzos/state.json` is the runtime journal: boot returns to the **last-active workspace** (a `BLITZ_WORKSPACE` pin overrides), and a boot after an unclean shutdown announces the crash to the human (chat line) + the agent (`trigger:'system'` moment, kind `crash`), enriched from macOS DiagnosticReports when possible. `markClean` runs LAST in `before-quit`/`gracefulExit` — "clean" means state flushed first. Two hosts on one root are detected via the journal's pid+heartbeat (loud warning today; refuse-enforcement pending decision). Tests: `node scripts/test-root-state.mjs`.
- **Effect-verified syscalls (`ok` = it happened + is durable):** `surface_control` click resolves the **visible, hit-testable** match (not the first `querySelector` hit — sites keep hidden twins; a selector matching only hidden/covered elements is a loud error), and click/type/key return an `effect` (typed value back, url/dom change) so the agent verifies in-band. Surface mutations are **authoritative-on-write**: main/serverOps apply create/update/move/close to `cached`/`osState` immediately (a `pendingCreates` set covers the pre-renderer-echo race), return `ok:false` for an unknown id (no more silent no-ops), and force a **durable flush at call time** on create/update/close so an `ok` ack survives a crash (the gap that lost a note). Tests: `node preview/test-control-effects.mjs` (needs CHROMIUM/Chrome).
- **Restart continuity / cross-workspace addressing** (item 4): surface ids already survive a restart (item 2's durable flush persists the workspace.json node at call time — verified). The remaining gap, **cross-workspace addressing**, is `findSurfaceWorkspace`/`relocateSurface` (workspace.mjs) + host `locateSurface`/`bringSurfaceHere`: an op on an id NOT in the active workspace no longer dead-ends — `move_surface` BRINGS the window into the active workspace (its content file is moved across folders, **id preserved**), while update/close/read return a navigable error naming the workspace ("move_surface it here, or switch_workspace"). The agent decides: just-this-window → move; whole-desktop → switch. Tests: `node scripts/test-cross-workspace.mjs`.
- **Guest capability contract** (`guest-capabilities.ts` + pure `popup-policy.mjs`): a web guest's browser-initiated escapes are handled by **what kind of action it is, never which site** (the old `accounts.google.com`/`contacts.google.com` popup regexes are deleted). Popups: sized `features`→real window (generalizes OAuth), `about:blank`→hidden child, link-disposition→new surface, else→deny+swallow the `top.location` hijack. Downloads→stream into the active workspace folder (→ a file tile). Permissions→real browser-parity **Allow/Block** prompt (the `consent-card` UI in App.tsx), remembered per-origin in the root journal (`getPermission`/`setPermission`); sensitive set prompts, harmless auto-allowed, unknown denied. `beforeunload`→auto-accept so close/nav always wins. Session-level handlers (download/permission) set once on `persist:agentos`; per-guest popup/unload in `did-attach-webview`. Tests: `node scripts/test-popup-policy.mjs`.
- **Onboarding P1 — the Case File board** (`plans/onboarding-case-file.md`): first launch (gated by `ONBOARDING_MODE` in `src/renderer/src/onboarding/config.ts`; `'always'` while iterating) shows a Dia-like opener (aurora + real scan progress, `OnboardingFlow.tsx`) that dissolves into a deterministic board — NO LLM. `src/main/onboarding.ts` (director) runs `scripts/onboarding-scan.mjs` as a child (`--json --progress` added; `@progress {json}` lines on stderr), creates + switches to the `case-file` workspace, and seeds template widgets per the **pure planner** `src/main/onboarding-board.mjs` (testable: `node scripts/test-onboarding-seed.mjs`). Board templates are ordinary library widgets — `profile`/`rhythm`/`toolbox`/`quotes`/`gaps`/`workflows` (new) + `dossiers`/`timeline` — registered in `widgets/widgets.json` (manifest-driven; an unlisted .html is invisible). When FDA is off, a native `unlock` card (runtime-only — listed in BOTH isRuntime predicates: store.ts applyReconcile + workspace-host.mjs; never a workspace node) deep-links System Settings; main polls the TCC probe and on grant re-scans + deepens the board in place. `.blitzos/onboarding/{scan.json,context.md,board.json}` in the workspace are the P2 brain's inputs (board.json maps card role→surface id). Onboarding never re-seeds over an existing board (cached path re-ensures only the unlock card). **Scan v2 + adaptive board:** sources now include Calendar, an AddressBook **contacts join** (handles→"First L."; unmatched stay hashed), a no-permission **document census** (mdfind), doc authors, and **web-first detection** (`scan.web.workflow` = curated SaaS matches w/ `integration` ids). The planner places **ON the home lattice** (no pixels): each card declares a CONTENT-driven span (`SIZE_FOR` — profile l, projects xl hero, rhythm l, workflows tall/l by row count…), `findSlot` resolves it against LIVE surfaces (the pinned chat hub included) with `near` composition hints; no fit → shrink (floor m) → park off-home below the frame. Web-first scans place Workflows (rows open sites via `blitz.tool open_window`) right after profile so it wins the prime spans; the unlock card is IN the plan when FDA is off (placement priority after gaps; `findUnlockSlot` covers the cached re-ensure). The board deliberately saturates past HOME_BUDGET — the brain's duty doc tells it to curate DOWN (`place_widget {id,size}` resize, `bring_home` parked cards, retire via `send_offscreen`, never close). The P2 brain's "import your Chrome life" conversation (open-tabs via Automation prompt, sign-in surfaces, OAuth action-items) is specced in the plan doc.
- **Onboarding P2/P3 — the primary agent carries interview, then initiative** (no separate runtime, no headless brain): the shared `agent-runtime.mjs` exposes a policy-free boot-task seam (`setBootTaskProvider` → `getBootTask`, an optional duty string re-read on every (re)launch and injected into the agent's bootstrap with an explicit license to act unprompted). index.ts wires `setBootTaskProvider((id) => id === '0' ? interviewBootTask() : null)`, so ONLY the primary agent (agent '0' — a visible managed agent terminal, owned by terminal-manager.mjs) gets the duty. `interviewBootTask()` returns the interview duty while `<ws>/.blitzos/onboarding/interview.json` is `pending`, then returns the resident initiative duty after it is `done`. The duty doc (`src/main/blitzos-interview.md` → copied to `.blitzos/onboarding/interview.md`) tells the agent: read `context.md` + `board.json`, ask ≤4 MC questions as ```blitz-ui choice cards in chat + 1 open voice sample, update board cards + flip gaps `done` per answer, finish with "What I learned" + `profile.md` + interview.json done, then propose initiatives and start one reversible initiative while staying within the user's act-vs-ask boundaries. **Launch branch** (`startInterviewPhase`): an available agent backend (`claude` by default, `codex` selectable) → `osKickBrain('0')`; no backend → a plain chat error says the real interviewer cannot start. There is intentionally no canned/static interview tier.
