# TCC + signing fixes tracker

Goal: no macOS permission prompt ever lands on the main Electron app (`dev.blitz.os`). Every TCC-gated op routes through the CU helper (`dev.blitz.os.computeruse`), which holds the grants; fall back to Electron only when the helper is absent. No agent doctrine change (all below the connection verb boundary). Detail docs: `plans/tcc-route-through-cu-plan.md` (routing), `plans/notarization-signing-handoff.md` (signing).

## Launch-prompt-spam saga — RESOLVED (2026-06-25)
The first-run "wall of permission dialogs" was THREE independent prompt classes, all rooted in two facts: **HOME is itself a git repo (`~/.git` dotfiles)** and the **hardened-runtime apps lacked the apple-events entitlement**. Found only by reading the live TCC log (`log show ... "Prompting"`), not by static analysis.
1. **Desktop/Documents/Downloads/Music folder prompts** ← the onboarding scan's `srcGit` `readdirSync`-walked HOME (its `walk()` recursed into protected subdirs). Fixed: **T0** (scan nuked to Branch B / fully gutted).
2. **Apple Events "requires entitlement" prompts** ← BlitzOS + the helper send Apple Events (Chrome-bridge: `countChromeWindows`, `openChromeJsRow`) under hardened runtime without `com.apple.security.automation.apple-events`; electron-builder's default entitlements omitted it and overwrote the helper's. Fixed: **X** (entitlements plists).
3. **Photos/Contacts/Calendar prompts** ← the interview agent's startup `git status` in `~/Blitz/Home` resolved the repo root to `~` (HOME repo) and walked the whole home dir. Fixed: **T2** (`git init ~/Blitz` = hard boundary; the env-var approach was too fragile through tmux).
Verified: a clean wipe+relaunch fires ZERO folder/Photos/Contacts/Calendar prompts; the only BlitzOS-charged prompt left is ONE `System Events` Automation consent (the normal, one-time kind, from the Chrome-bridge step). Eliminating that = the routing batch (C/PG: route through the pre-granted helper, or skip Chrome detection when the default browser isn't Chrome). Separate, not BlitzOS: the user's own `Codex.app` walks the HOME repo too.

## Apple-Events class — CLOSED (full src/main sweep, 2026-06-25)
Every osascript / Apple-Events EXECUTION site in `src/main` is now either helper-routed (direct execFile only as the helper-absent fallback) or deleted. Exhaustive list:
- `connection-safari-link.mjs` / `connection-chrome-applescript-link.mjs` — single `osa()` → `helper.call('osa')`, execFile fallback. (Safari/Chrome tab list + drive.)
- `index.ts` `matchChromeTabByBounds` — `helper.call('osa')`, execFile fallback. (Chrome-window drop → active-tab bridge.)
- `blitz-chrome.ts` foreground — `helper.call('activate')`, System-Events osascript fallback. (Blitz Chrome only.)
- `onboarding.ts` (`openChromeJsRow`, `requestHelperAutomation`, the View▸Developer menu-drive, the JS-enabled probe) — all via `helper.runScan(...)` so the grant lands on the helper.
- `onboarding.ts` `requestAutomation` + `preboardTabsPath` + its IPC/preload — DELETED.
Net: with the helper present (the normal post-onboarding state) NO Electron-direct Apple Event fires, so no prompt is ever attributed to `dev.blitz.os`. The fallbacks exist only for a degraded run with no helper.
